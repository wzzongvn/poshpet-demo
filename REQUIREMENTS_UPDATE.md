# Part 1. 2026-05 검수 (R1~R9)

## R1. 고양이 광견병 백신 항목 제거
- **요건**: 반려동물 종이 **고양이**이면 백신 입력/표시에서 **광견병(Rabies) 항목을 노출하지 않는다.** 종합백신(FVRCP)만 묻는다.
- **이유**: 매장 운영 기준상 고양이는 광견병 미대상.
- **개발 시 주의**: 백신 상태 판정 로직(`vaccinesSummary`)도 고양이는 rabies를 판정에서 제외해야 한다. (개=rabies+core 둘 다, 고양이=core만)
- **영향 범위**: 펫 등록/수정 시트, 펫 상세 카드, 예약 확인 백신 안내, 관리자 백신 알림.

## R2. 로그인 / 회원가입 UI 분리
- **요건**: "로그인"과 "회원가입(간편 가입)"의 화면이 지금은 동일하다. **두 화면의 UI/문구를 구분**한다.
  - 로그인: 기존 회원이 다시 들어오는 맥락
  - 가입: 신규 회원
- **개발 시**: 진입점(`openAuth`)에서 mode('login'|'signup')를 구분해 타이틀/리드/CTA 문구를 다르게.

## R3. 로그인 수단 기억 (default 표시)
- **요건**: 한번 어떤 방법(Kakao/Zalo/Google/전화)으로 로그인하면, 다음부터 **그 수단을 기본(default)으로 크게** 보여주고, 나머지는 "다른 방법으로 로그인" 형태로 접어둔다.
- **개발 시**: 마지막 사용 provider를 저장(localStorage 또는 서버 프로필).

## R4. 픽업 옵션 기본값 = "네, 픽업 받을게요"
- **요건**: 예약 흐름의 픽업 단계에서 기본 선택을 **픽업 받기(yes)**로 한다.

## R5. 예약 확정 후 "예약 수정" 바로가기 버튼

- **개발 시**: 기존 `changeBooking(id)` 로직 재사용.

## R6. 예약 버튼 진입 분기 (기존 예약 있으면 최근 예약 먼저)

# Part 2. 2026-05-29 추가 검수 (R10~R16)

> 이번 검수는 대부분 **iOS Safari 실기기(아이폰 16 Pro Max)** 에서만 재현된 문제다. PC/시뮬레이션으로는 안 보였다.
> **→ 실개발 시 iOS Safari 실기기 검수 필수.** 특히 한글 입력(IME), 큰 글씨 접근성 설정, 고정 푸터.
>
> 또한 "고쳤는데 화면이 그대로"가 반복됐는데 원인은 **배포 미반영/캐시**였다.
> → 빌드 버전 표식을 화면 좌하단에 상시 노출해 즉시 확인.

## R10. 예약 달력 — iOS '더 큰 텍스트'에서 마지막 주 가림
- **증상**: 월말(예: 5/29) 진입 시 선택 가능한 날(29·30·31)이 달력 마지막 줄에 몰리는데, iOS "더 큰 텍스트(Dynamic Type)" + "볼드체 텍스트"를 켜면 글씨·줄높이가 부풀어 마지막 줄이 고정 버튼에 가려 **선택 가능한 날이 화면에 하나도 안 보임**.
- **근본 원인**: 페이지에 `text-size-adjust` 설정이 없어, iOS가 OS 글씨 설정대로 웹 폰트를 강제 확대.
- **해결**:
  1. `html{-webkit-text-size-adjust:100%; text-size-adjust:100%;}` 추가 (핵심)
  2. 달력 셀 `.day`를 `aspect-ratio` → **고정 높이 40px**로
  3. 날짜 단계 하단 spacer 170→210px
- **개발 시 주의**: 모바일 웹 루트에 `text-size-adjust:100%` 반드시. 달력 같은 그리드는 폰트 독립 고정 셀 높이.

## R11. 픽업 단계 — 주소 검증 로직 모순으로 '예약 확인' 버튼 영구 비활성
- **증상**: 픽업="네"에서 별칭+상세주소를 다 입력해도 버튼이 회색.
- **근본 원인**: 입력칸 표시 조건과 검증 조건의 **불일치**.
  - 표시(`pickupAddrSection`): 저장 주소 0건이면 입력칸을 띄움
  - 검증(`addressReady`): `addrId==='__new__'`일 때만 입력값 검증 → 0건일 땐 `addrId`가 `null`이라 분기 못 타고 **항상 false**
- **해결**: 검증의 입력모드 판정을 표시 조건과 일치 → `isInputMode = addrId==='__new__' || addresses.length===0`. 전화(`phoneReady`)도 동일 통일.
- **개발 시 원칙(중요)**: **"입력칸을 띄우는 조건"과 "그 입력값을 검증하는 조건"은 반드시 단일 진실로 일치**시킬 것.

## R12. 픽업 단계 — 연락처 기본값 자동 선택
- **증상**: 진입 시 주소는 자동 선택되나 연락받을 번호가 미선택이라 `phoneReady()`가 false → 버튼 비활성.
- **해결**: 픽업 진입 시 저장된 기본 번호(`isDefault`) 또는 첫 번호 자동 선택. 주소도 동일.

## R13. 한글 입력(IME) — 조합 중 버튼/검증 미반영
- **증상**: 한글 입력 시 마지막 글자가 조합 중이면 `oninput`이 미완성 글자로만 발화되거나 늦게 발화돼, 입력을 끝내도 버튼이 활성화 안 됨. **영어로는 정상**(=IME 문제 확정).
- **해결**: 픽업 입력칸 4종(주소 별칭/상세, 전화 라벨/번호) + 펫 이름 입력칸에 **`oninput`·`onchange`·`onblur`·`oncompositionend` 4종 이벤트** 연결. 공통 핸들러 `syncPickupInputs()`가 각 input 값을 직접 읽어 상태 반영 후 버튼 갱신.
- **영향 범위**: 한글·베트남어 등 IME 사용 언어 전반.
- **개발 시 원칙(중요)**: 한국·베트남 고객 대상이므로, **버튼 활성/폼 검증에 관여하는 모든 텍스트 입력**은 `oninput`만 의존 금지. composition 이벤트 처리 또는 blur/change 보강. (React: `onCompositionEnd` 또는 controlled input + composition 플래그)

## R14. 픽업 단계 — '예약 확인' 버튼 위치(입력 요소 가림)
- **증상**: 버튼이 `position:absolute; bottom:74px` 고정이라, 입력 요소가 많은 픽업 단계에서 콘텐츠 중간(상세주소와 전화번호 사이)에 떠서 안내문·전화번호 입력을 덮음.
- **해결**: `flowFoot(enabled, label, inline)`에 inline 모드 추가. 픽업 단계만 `inline=true`로 → `.flowfoot.inline{position:static}`으로 **콘텐츠 흐름 맨 아래(주소·전화·메모 다음)**에 배치. 다른 단계는 기존 고정 유지.
- **개발 시 원칙**: 입력 폼이 길어질 수 있는 단계는 sticky 푸터 대신 흐름 배치 또는 충분한 하단 패딩.

## R15. 예약 단계 진행 불가 — refreshFootBtn 오작동(부작용 회귀)
- **증상**: 메뉴/날짜 단계에서 선택해도 다음 버튼이 회색이라 진행 불가.
- **근본 원인**: 픽업용 버튼 갱신 함수 `refreshFootBtn()`(주소·전화 검증)을 `bindFlow`가 **모든 단계에서 무조건 호출** → 주소·전화가 비어있는 메뉴/날짜에서 버튼을 강제 비활성화.
- **해결**: `bindFlow`의 `refreshFootBtn()` 호출을 **`curStep()==='pickup'`일 때만** 실행.
- **개발 시 교훈**: 특정 단계 전용 갱신 로직을 전역 lifecycle 훅(bindFlow/useEffect)에 넣을 때는 **반드시 단계 조건 가드**.

## R16. 픽업 주소 — 첫 주소가 저장되지 않던 문제 (전화번호와 동작 불일치)
- **요건**: 저장된 주소가 없으면 입력칸, **하나라도 저장되면 다음 예약부터 저장 주소 우선 자동 선택 + 목록 + "다른 주소 추가"** (전화번호와 동일).
- **근본 원인**: 예약 저장(`submitBooking`)이 주소를 `addrId==='__new__'`일 때만 저장. 데모는 주소 0건에서 시작해 입력칸이 바로 떠도 `addrId`가 `null`이라 **첫 주소가 영영 저장 안 됨**. (전화는 "본인" 기본번호가 항상 있어 정상 저장됐던 것.)
- **해결**:
  - 저장 조건을 `addrInputMode = (addrId==='__new__') || (addresses.length===0)`로 확장 → 첫 주소도 저장
  - 저장 후 임시 입력값(addonAddr/addonAddrLabel) 정리 → 다음 예약 시 저장 주소 우선 선택
  - 진입 시 자동선택을 `pickup` 상태 무관하게(저장 주소 있고 미선택이면) 처리
- **검증된 흐름**: 1차(0건)→입력→저장 / 2차(1건)→자동선택+목록 / "다른 주소 추가"→입력칸 전환
- **개발 시 원칙**: 주소·전화 등 "저장 후 재사용" 항목은 **0건→입력→저장→자동선택→추가** 흐름을 동일 패턴으로.

## 실개발 필수 반영 원칙 (R10~R16에서 도출)

1. **모바일 웹 루트에 `text-size-adjust:100%`** — OS 글씨 확대로 레이아웃 깨짐 방지. 그리드(달력 등)는 폰트 독립 고정 셀 높이.
2. **IME(한글·베트남어) 입력 처리** — 버튼 활성/폼 검증에 관여하는 텍스트 입력은 `oninput`만 의존 금지. composition 이벤트 또는 blur/change 보강.
3. **"입력칸 표시 조건"과 "검증 조건"은 단일 진실로 일치** — 둘이 어긋나면 "입력칸은 보이는데 버튼이 안 켜지는" 모순.
4. **단계 전용 로직을 전역 lifecycle 훅에 넣을 때 단계 조건 가드 필수**.
5. **긴 입력 폼 단계는 sticky 푸터 대신 흐름 배치/충분한 하단 패딩**.
6. **주소·전화 등 "저장 후 재사용" 항목은 0건→입력→저장→자동선택→추가 흐름을 동일 패턴으로** 구현(저장 누락 주의).
7. **배포 반영 확인용 빌드 버전 표식**(`build MM-DD HH:MM`)을 개발 환경에 상시 노출 권장.
