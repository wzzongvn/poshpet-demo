> 이 문서는 2026-05-29 사장님 데모 검수에서 발견된 이슈와 그 해결 내역이다.
> 앞선 R1~R9(REQUIREMENTS_UPDATE.md)에 이어지는 확정 요건이며,
> **프로토타입(index.html)뿐 아니라 향후 실제 개발(Next.js 본 사이트)에도 동일하게 반영**되어야 한다.

## 배경 — 이번 세션에서 드러난 핵심 교훈

이번 검수의 상당수는 **iOS Safari 실기기(아이폰 16 Pro Max)** 에서만 재현되는 문제였다.
PC 브라우저나 코드 시뮬레이션으로는 안 보이고, 실제 폰에서만 깨졌다.
**→ 실개발 시 반드시 iOS Safari 실기기로 검수할 것. 특히 한글 입력(IME), 큰 글씨 접근성 설정, 고정 푸터.**

또한 "고쳤는데 화면이 그대로"인 상황이 반복됐는데, 원인은 **배포 미반영/캐시**였다.
**→ 빌드 버전 표식(R16)을 상시 노출해, 화면에서 즉시 배포 반영 여부를 확인할 수 있게 함.**

## R10. 예약 달력 — iOS '더 큰 텍스트'에서 마지막 주(週) 가림
- **증상**: 월말(예: 5/29) 진입 시 선택 가능한 날(29·30·31)이 달력 마지막 줄에 몰리는데, iOS "더 큰 텍스트(Dynamic Type)" + "볼드체 텍스트"를 켜면 글씨·줄높이가 부풀어 마지막 줄이 고정 버튼("시간 선택")에 가려 **선택 가능한 날이 화면에 하나도 안 보임**.
- **근본 원인**: 페이지에 `text-size-adjust` 설정이 없어, iOS가 OS 글씨 크기 설정대로 웹 폰트를 강제 확대.
- **해결(3중)**:
  1. `html{-webkit-text-size-adjust:100%; text-size-adjust:100%;}` — OS 자동 글씨 확대 차단(핵심).
  2. 달력 셀 `.day`를 `aspect-ratio` → **고정 높이 40px**로. 폰트가 커져도 6주 높이 일정.
  3. 날짜 단계 하단 spacer 170→210px (추가 방어).
- **실개발 주의**: 모바일 웹은 `text-size-adjust:100%`를 루트에 반드시 둘 것. 달력 같은 그리드는 폰트 크기에 독립적인 고정 셀 높이 사용.

## R11. 픽업 단계 — 주소 검증 로직 모순으로 '예약 확인' 버튼 영구 비활성
- **증상**: 픽업="네"에서 주소 별칭+상세주소를 모두 입력해도 "예약 확인" 버튼이 회색(비활성) 그대로.
- **근본 원인**: 입력칸 표시 조건과 검증 조건의 **불일치**.
  - 표시(`pickupAddrSection`): 저장 주소 0건이면 입력칸을 띄움.
  - 검증(`addressReady`): `addrId==='__new__'`일 때만 입력값 검증 → 0건일 땐 `addrId`가 `null`이라 이 분기를 못 타고, 저장주소를 찾으려다 **항상 false**.
- **해결**: 검증의 입력모드 판정을 표시 조건과 일치 → `isInputMode = addrId==='__new__' || addresses.length===0`. 전화번호(`phoneReady`)도 동일 원칙으로 통일.
- **실개발 원칙**: **"입력칸을 띄우는 조건"과 "그 입력값을 검증하는 조건"은 반드시 단일 진실(single source)로 일치**시킬 것.

## R12. 픽업 단계 — 연락처 기본값 자동 선택
- **증상**: 진입 시 주소는 자동 선택되나 연락받을 번호가 미선택이라 `phoneReady()`가 false → 버튼 비활성.
- **해결**: 픽업 진입 시 저장된 기본 번호(`isDefault`) 또는 첫 번호를 자동 선택. 주소도 동일.

## R13. 한글 입력(IME) — 조합 중 버튼/검증 미반영
- **증상**: 한글로 입력 시(예: 별칭 "우리집") 마지막 글자가 조합 중이면 `oninput`이 미완성 글자로만 발화되거나 늦게 발화돼, 입력을 끝내도 버튼이 활성화 안 됨. **영어로는 정상 작동**(=IME 문제 확정).
- **해결**: 픽업 입력칸 4종(주소 별칭/상세, 전화 라벨/번호) + 펫 이름 입력칸에 **`oninput`·`onchange`·`onblur`·`oncompositionend` 4종 이벤트** 연결. 공통 핸들러 `syncPickupInputs()`가 각 input 값을 직접 읽어 상태 반영 후 버튼 갱신.
- **영향 범위**: 한글/베트남어 등 IME 사용 언어 전반.
- **실개발 원칙(중요)**: 한국·베트남 고객 대상 서비스이므로, **버튼 활성/폼 검증에 관여하는 모든 텍스트 입력**은 `oninput`만 의존하지 말 것. `compositionstart/compositionend`를 처리하거나, blur/change로 보강할 것. (React라면 `onCompositionEnd` 핸들링 또는 controlled input + composition 플래그.)

## R14. 픽업 단계 — '예약 확인' 버튼 위치(입력 요소 가림)
- **증상**: 버튼이 `position:absolute; bottom:74px` 고정이라, 입력 요소가 많은 픽업 단계에서 버튼이 콘텐츠 중간(상세주소와 전화번호 사이)에 떠서 안내문·전화번호 입력을 덮음.
- **해결**: `flowFoot(enabled, label, inline)`에 inline 모드 추가. 픽업 단계만 `inline=true`로 호출 → `.flowfoot.inline{position:static}`로 **콘텐츠 흐름 맨 아래(주소·전화·메모 다음)**에 배치. 다른 단계(룸·날짜·시간·펫)는 콘텐츠가 짧아 기존 고정 유지.
- **실개발 원칙**: 입력 폼이 길어질 수 있는 단계는 sticky 푸터 대신 흐름 배치 또는 충분한 하단 패딩 확보.

## R15. 예약 단계 진행 불가 — refreshFootBtn 오작동(부작용 회귀)
- **증상**: 메뉴/날짜 단계에서 선택을 해도 다음 버튼이 회색이라 진행 불가.
- **근본 원인**: 픽업용 버튼 갱신 함수 `refreshFootBtn()`(주소·전화 검증)을 `bindFlow`가 **모든 단계에서 무조건 호출** → 주소·전화가 비어있는 메뉴/날짜 단계에서 버튼을 강제 비활성화.
- **해결**: `bindFlow`의 `refreshFootBtn()` 호출을 **`curStep()==='pickup'`일 때만** 실행하도록 한정(try/catch 보호).
- **실개발 교훈**: 특정 단계 전용 갱신 로직을 전역 lifecycle 훅(bindFlow/useEffect)에 넣을 때는 **반드시 단계 조건으로 가드**할 것.

## R16. 픽업 주소 — 첫 주소가 저장되지 않던 문제 (전화번호와 동작 불일치)
- **요건**: 저장된 주소가 없으면 입력칸을 띄우고, **하나라도 저장되면 다음 예약부터 저장 주소를 우선 자동 선택 + 목록에서 선택 + "다른 주소 추가"로 추가 입력** 가능해야 한다. (전화번호는 이미 이렇게 동작 중 → 주소도 동일하게.)
- **근본 원인**: 예약 저장(`submitBooking`)이 주소를 `addrId==='__new__'`일 때만 저장. 데모는 주소 0건에서 시작해 입력칸이 바로 떠도 `addrId`가 `null`(=`__new__` 아님)이라 **첫 주소가 영영 저장 안 됨**. (전화는 "본인" 기본번호가 항상 있어 "다른 번호 추가" 시 `__new__`가 되므로 정상 저장됐던 것.)
- **해결**:
  - 저장 조건을 `addrInputMode = (addrId==='__new__') || (addresses.length===0)` 로 확장 → 첫 주소도 저장.
  - 저장 후 임시 입력값(addonAddr/addonAddrLabel) 정리 → 다음 예약 시 저장 주소가 우선 선택되도록.
  - 진입 시 자동선택을 `pickup` 상태와 무관하게(저장 주소 있고 미선택이면) 처리 — 전화번호와 동일 루틴.
- **검증된 흐름**: 1차(0건)→입력→저장 / 2차(1건)→자동선택+목록 / "다른 주소 추가"→입력칸 전환.

## R(부가). 빌드 버전 표식
- 화면 **좌하단**(하단 탭바 바로 위)에 `build MM-DD HH:MM` 갈색 칩 상시 노출. `BUILD_VERSION` 상수 1곳에서 관리, 부팅 시 `#buildBadge`에 주입.
- 목적: 배포·캐시 반영 여부를 화면에서 즉시 확인(개발 중). 클릭 통과(`pointer-events:none`)라 조작 방해 없음.
- **실개발**: 스테이징/개발 환경에서만 노출하거나, 빌드 타임에 커밋 해시·시각 자동 주입 권장.
